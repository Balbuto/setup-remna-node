# 🤝 Руководство по содействию (CONTRIBUTING.md)

Мы рады вашему интересу к проекту **RemnaNode Advanced Installer**! Данный проект развивается силами сообщества, и мы приветствуем любые улучшения, исправления багов и новые функции.

Пожалуйста, ознакомьтесь с этим руководством перед отправкой Pull Request (PR) или созданием Issue.

---

## 🐛 Как сообщить об ошибке (Report a Bug)

1. Убедитесь, что вы используете последнюю версию скрипта. Выполните команду `remnanode` и проверьте, нет ли обновлений.
2. Проверьте вкладку **Issues** в репозитории на предмет похожих обращений.
3. Если ошибка уникальна, создайте новый Issue и укажите:
   * Версию ОС вашего сервера (`cat /etc/os-release`).
   * Архитектуру процессора (`uname -m`).
   * Подробные шаги для воспроизведения проблемы.
   * Выдержку из файла логов установки `/var/log/remnanode_install.log` (без конфиденциальных токенов и паролей).

Если у вас есть идея, как сделать скрипт еще лучше:

3. Будьте готовы обсудить детали реализации с мейнтейнерами.

## 💻 Требования к коду (Code Standards)

Если вы хотите внести изменения в код скрипта `setup_node.sh`, пожалуйста, придерживайтесь следующих жестких стандартов:

### 1. Строгий режим Bash (Bash Strict Mode)
Все сценарии должны работать в безопасном режиме во избежание непредсказуемого поведения ядра ОС:
```bash
set -Eeuo pipefail
IFS=$'\n\t'
```
* Любое объявление функций или переменных должно быть безопасным по отношению к пустому значению (unbound variables).

### 2. Проверка через ShellCheck
Перед отправкой изменений обязательно пропустите скрипт через статический анализатор **ShellCheck**:
```bash
shellcheck setup_node.sh
```
В коде не должно быть критических замечаний и предупреждений (errors & warnings).

### 3. Стиль кодирования (Code Style)
* **Язык сообщений**: Все диалоговые окна, сообщения об ошибках, логи и отчеты должны выводиться на **русском языке**.
* **Именование переменных**: Используйте `local` переменные внутри функций. Названия переменных должны быть в `snake_case` (например, `local panel_ip`), а глобальные константы — в `UPPER_CASE` (например, `APP_DIR`).
* **Отступы**: Используйте 4 пробела для отступов (не табы).

## 🚀 Процесс отправки Pull Request (PR)

1. Сделайте форк (Fork) официального репозитория.
2. Создайте свою ветку для фичи или исправления:
   ```bash
   git checkout -b feature/amazing-new-option
   ```
3. Внесите изменения и протестируйте их на чистой виртуальной машине (KVM Ubuntu/Debian).
4. Зафиксируйте коммиты с понятными описаниями:
   ```bash
   git commit -m "Fix: исправление парсинга портов UFW"
   ```
5. Отправьте ветку в свой форк:
   ```bash
   git push origin feature/amazing-new-option
   ```
6. Откройте Pull Request в ветку `main` оригинального репозитория.

Спасибо за ваш вклад в развитие свободных инструментов приватности! 🌍
